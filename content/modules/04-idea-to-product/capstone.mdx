## The Project

The OMDb Movie Explorer is a real, deployed web app for discovering and tracking movies and TV shows. Search any title, view full details, add titles to a watchlist with "To Watch" and "Watched" sections, and get AI-powered summaries. It was built using Claude Code and the GSD workflow — zero manual code written. This is the proof-of-mastery project: every skill from every module is applied to ship something real.

**Live app:** [https://omdb-explorer-theta.vercel.app](https://omdb-explorer-theta.vercel.app)

## What Was Built

- A live web app at [https://omdb-explorer-theta.vercel.app](https://omdb-explorer-theta.vercel.app) with full movie and show search
- Search powered by the OMDb API (Open Movie Database) — hundreds of thousands of titles
- Detail modal with director, cast, plot, runtime, genre, and IMDb rating (raw "N/A" fields hidden via omdbVal() guard pattern)
- Watchlist with "To Watch" / "Watched" sections persisted in localStorage with hydration guard
- Dark cinematic theme (zinc-950 background, amber accent) using shadcn/ui and Tailwind v4
- Server-side API key proxying via Next.js API routes — API key never exposed to the browser
- Deployed to Vercel with OMDB_API_KEY configured in production environment variables
- A GitHub repo with a clean commit history built via the GSD workflow

## The Problem Being Solved

Finding and tracking movies and shows requires juggling multiple apps: one for search, one to check ratings, another to save your list. The OMDb Movie Explorer combines search, detail lookup, and watchlist tracking in a single fast-loading, dark-mode interface. No accounts required. No subscription. Opens in 2 seconds and remembers your watchlist across sessions via localStorage. Real value for anyone who watches movies and wants a personal tracking tool they actually built themselves.

## Retrospective

The OMDb Movie Explorer demonstrated three things clearly:

First, the GSD methodology works for real products. Breaking the build into three plans — scaffold, features, deploy — meant each phase had a clear done condition. No phase was "done when it feels complete." The done condition was defined before building.

Second, server-side API key proxying is a pattern worth knowing by default. Every app that uses a third-party API with a secret key should proxy through an API route. Claude Code implemented this correctly the first time when given the explicit constraint.

Third, localStorage with a hydration guard is the right persistence approach for personal tools that don't need user accounts. The pattern (read in useEffect, not at render time, with a `hydrated` flag) prevented SSR hydration mismatches and produced a working implementation on the first build.

The meta-lesson: constraints produce better software. Specifying "no database, no accounts, no NEXT_PUBLIC_ prefix for the API key" resulted in a simpler, more correct architecture than leaving those decisions open.
