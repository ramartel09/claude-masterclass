**The capstone project: OMDb Movie Explorer**

The capstone project for this course is the tool we built through Phase 4: the OMDb Movie Explorer — a fully deployed, personal movie and TV show search tool with search, a watchlist, and AI-powered summaries.

**What was built**

The OMDb Movie Explorer is a Next.js web application that does three things well:

*Search:* Type any movie or TV show title into the search bar and get a grid of results with poster images, titles, years, and media types. Results come from the OMDb API (the Open Movie Database), which has data on hundreds of thousands of titles. The search is debounced so it doesn't fire on every keystroke — it waits for you to pause.

*Detail modal:* Click any result card and a modal opens with the full details: director, cast, plot summary, runtime, genre, IMDb rating, and a higher-resolution poster. Fields that aren't available are gracefully hidden rather than showing raw "N/A" text — a small detail that makes the app feel polished.

*Watchlist with persistence:* Add any title to your watchlist with one click. The watchlist has two sections: "To Watch" and "Watched." Toggle a title between the two sections as you work through your backlog. Everything persists in localStorage — close the tab, come back a week later, your watchlist is still there.

**How it was built**

The OMDb Movie Explorer was built using exactly the methodology this course teaches:

Phase 4 Plan 01 scaffolded the Next.js 16 project with TypeScript, Tailwind CSS v4, and shadcn/ui. The scaffold included the basic project structure — app directory, components, lib utilities — but no features yet. The scaffold was verified running before any feature code was written.

Phase 4 Plan 02 built all the features: the search API route that proxies the OMDb API (keeping the API key server-side, never exposed to the browser), the detail API route for individual title lookups, the search results grid using shadcn/ui Card components, the detail Dialog modal, and the watchlist page with localStorage persistence. Every component was built from a plain-English description of what it should look like and what data it should show.

Phase 4 Plan 03 (this plan) deployed the application to Vercel, set the OMDB_API_KEY in production environment variables, and verified the full feature set works on the live URL.

**The key technical decisions and why they matter**

*Server-side API key proxying:* The OMDb API requires an API key. Instead of putting that key in the browser (where anyone could see it in the network tab), the app has two API routes — `/api/search` and `/api/detail` — that make the OMDb requests server-side and return the results to the browser. The key never leaves the server. This is the right pattern for any app that uses a third-party API with a secret key.

*localStorage persistence with a hydration guard:* The watchlist uses localStorage so it persists across browser sessions without needing a database or user accounts. The tricky part with localStorage in Next.js is that the server doesn't have access to it — attempting to read localStorage during server-side rendering causes a hydration mismatch error. The solution is a `hydrated` state flag: the component renders an empty state on the server, then reads localStorage inside a `useEffect` after the component mounts on the client. This pattern appears throughout the app and is worth understanding — it applies to any client-side persistence with Next.js.

*shadcn/ui component composition:* The entire UI is built from shadcn/ui primitives: Card, Dialog, Button, Badge, Input, Tabs. No custom CSS beyond Tailwind utilities. This is the constraint library approach from Lesson 6: choose a component library, use its vocabulary, and let Claude Code fill in the details within those constraints. The result is consistent, polished, and maintainable.

*omdbVal() guard pattern:* The OMDb API returns "N/A" as a string for missing fields, not null or undefined. A utility function `omdbVal()` checks for this and returns an empty string instead — so conditional rendering (`{omdbVal(movie.Director) && ...}`) works correctly. This is a small pattern that eliminated every instance of raw "N/A" text in the UI.

## Try It

**Task:** Extend the OMDb Movie Explorer with one new feature. Pick one of the options below, or propose your own.

**Option A — Favorites filter:** On the watchlist page, add a "Favorites" section alongside "To Watch" and "Watched." Add a favorites toggle (a star icon) to each watchlist entry. Favorites are stored in the same localStorage structure.

**Option B — User ratings:** Add a 1–5 star rating field to each watchlist entry. Display the rating on the watchlist card. Store it in localStorage alongside the other watchlist data.

**Option C — Curated list:** Search the live app for your personal "Best of 2024" picks. Add 10 titles to your watchlist, mark them all as "Watched," and screenshot the result. This is a usage exercise, not a code exercise — but it tests all three features (search, detail, watchlist).
