**The final capstone: GTA Hospital Wait Time Tracker**

The capstone project for the entire course is the tool we've been building toward through Module 4: a web app that shows current estimated emergency room wait times at major GTA hospitals.

**Why this project**

The hospital tracker was chosen as the capstone for three reasons:

*It's a real problem.* Anyone who has needed emergency care in the GTA has experienced the uncertainty of not knowing which hospital has a shorter wait. The product solves this — or it tries to, which is the right ambition for a solo builder.

*It tests all four modules.* Prompting skills (writing clear specifications for Claude Code), Claude Code and GSD (the build methodology), agent design (the data-gathering component), and the idea-to-product pipeline (requirements, roadmap, scaffold, design, deploy). Every lesson shows up in this project.

*It has real constraints.* Public health data availability in Ontario is a genuine uncertainty. The PHIPA (Personal Health Information Protection Act) implications of health data are real. The data sources are real and imperfect. These constraints make it a better learning experience than a project with perfectly clean, freely available data.

**What Phase 4 will build**

Phase 4 (the final phase of the masterclass) builds the hospital tracker. Specifically:

- A Next.js web app deployed to Vercel with a live public URL
- Displays current estimated wait times for the six largest GTA emergency departments
- Color-coded wait status (green/yellow/red) based on wait time ranges
- Mobile-responsive layout readable on a phone
- Data source: Ontario's publicly available health data (CIHI, Ontario Health, or similar aggregate sources)
- Static data updated manually or on a schedule (no real-time API in v1)

**What "mastery" looks like when this is done**

When Phase 4 is complete and the hospital tracker is live, you'll have built a real web application that:
- Solves a real problem for a real user
- Was built in days, not months
- Required no writing of code by you
- Was deployed to the internet, accessible from any device
- Used the exact methodology this course teaches — the meta-demonstration completed

That's mastery. Not an abstract credential — a concrete artifact that demonstrates capability. You have the URL. The app works. You built it.

**Preview of Phase 4**

Phase 4 will pick up from your Phase 1–3 skills and apply them to a defined brief. The Phase 4 PLAN.md will specify exactly what to build in what order. You'll run `/gsd:execute-phase` and watch the tracker come together.

The Phase 4 plan includes: data source research and validation, hospital data model design, wait time display component, filtering and ranking, deployment and testing on real devices.

One note of caution from the research phase of this project: the GTA hospital data source is listed as MEDIUM confidence in the project planning. Real-time ER wait data from Ontario public health is not available through a clean public API as of this writing. Phase 4 may use estimated/approximate data, aggregate statistics from CIHI, or simulated data in a format that matches real data structure. The engineering work is the same; the data may be approximated.
